5. Critical Bugs
CRITICAL
Metrics Race Condition — Dashboard Goes Blank Every Minute
/api/collector deletes ALL rows from office_system_metrics then inserts one fresh row. The /api/metrics route uses .single() which returns a Supabase PGRST116 error when the table is momentarily empty. The useMetrics hook has no fallbackData, so during that ~10ms window the CPU/RAM bars on the dashboard blank out or flash an error state on every collector run (60 times/hour).
Fix: Replace DELETE+INSERT with .upsert({ id: 1, ...body.system, updated_at: now }, { onConflict: 'id' }) using a fixed row ID. Requires adding an id column (e.g., serial or constant 1) to the table, or using any stable unique field like updated_at is already used but wrong for conflict resolution. Alternatively, use fallbackData: defaultMetrics in useMetrics as a short-term mitigation.
CRITICAL
/api/restart and /api/command Are Completely Non-Functional
restart: COLLECTOR_INTERNAL_URL is not set in .env.local. Every POST to /api/restart returns 503. The Restart button in DashboardPanel has never worked in production.

command: The office_commands table receives inserts but nothing ever reads from it. There is no polling loop in the collector or any other script. All 4 allowed commands (restart_gateway, restart_userbot, run_collector, clear_cache) are dead letters.
Fix (restart): Add COLLECTOR_INTERNAL_URL=http://localhost:PORT to .env.local, expose an HTTP endpoint on the collector that accepts restart commands.
Fix (command): Add a poll loop to agent-metrics-collector.sh that queries office_commands WHERE status = 'pending', executes the command, and updates status to executed.
CRITICAL
AI Studio — FAL_KEY Missing, Model Label/API Mismatch
FAL_KEY is not in .env.local (or Vercel env vars). Every image generation request will fail with Authorization: Key undefined. The error from fal.ai (401) propagates to the UI as a generic error string.

Additionally, the AIStudioPanel displays "FLUX.1 Schnell FP8" as the model name but the /api/generate route calls fal-ai/flux-pro/v1.1-ultra — a completely different, more expensive model. These are not the same model; if FAL_KEY were present, costs would be higher than the UI implies.
Fix: Add FAL_KEY=... to .env.local and to Vercel environment variables. Align the model IDs — either update the API route to use fal-ai/flux/schnell or update the MODELS array to show "FLUX Pro v1.1 Ultra".
CRITICAL
PATCH /api/tasks Accepts Arbitrary Column Overwrites
The PATCH handler does const { id, ...updates } = body and passes all remaining fields directly to .update(updates). An attacker (or a bug in Natali's webhook code) can overwrite any column including created_by, created_at, position, or even inject SQL-unsafe values if Supabase ever relaxes its type checking. This is an injection surface even from trusted callers making mistakes.
Fix: Whitelist the allowed update fields: const allowed = ['title','description','status','priority','project','assigned_to','position','moved_by']; const updates = Object.fromEntries(Object.entries(body).filter(([k]) => allowed.includes(k)));

10 Design Studio Agents Never Show Real Status
STATIC_AGENTS defines 10 Design Studio agents (viktor, pixel, marka, sait, ruh, prostir, systema, koddesign, kontenta, doslidnyk) but parse-agent-sessions.py only processes 15 legacy agent IDs. HIGH
import-agent-history.py Has Silent Env Var Bug
The script reads os.environ.get('SUPABASE_URL') and os.environ.get('SUPABASE_KEY'). The calling shell script exports them as:
SUPABASE_URL=$(grep NEXT_PUBLIC_SUPABASE_URL /Users/a1/Projects/.../env.local | cut -d= -f2) SUPABASE_KEY=$(grep SUPABASE_SERVICE_ROLE_KEY /Users/a1/Projects/.../env.local | cut -d= -f2)
If the path is wrong or grep returns empty, the script prints "Missing SUPABASE_URL or SUPABASE_KEY" to stdout (which is redirected to /dev/null) and silently does nothing. Agent history from Claude Code subagent files never gets imported, so agent session history on the dashboard may be stale or empty.
Fix: Add error logging to stderr, use 2>&1 | tee /tmp/import-agent-history.log in the cron call, and validate env vars at the shell level before calling the Python script.
HIGH
Synthetic Heartbeat Noise Pollutes Activity Feed
When no agents are working, the collector inserts one of 5 canned messages into office_activity_log every minute (rotated by minute % 5). Examples: "All agents on standby — ready for tasks", "System heartbeat — all services nominal". These are indistinguishable from real events in the activity feed. Dan cannot tell from the dashboard whether Natali actually took an action or whether it's just the heartbeat. The table fills with ~288 synthetic rows per hour during idle periods.
Fix: Add a type column to office_activity_log (or use a separate office_heartbeats table). Filter synthetic events out of the /api/activity response, or at minimum render them differently in the UI (dimmed, no icon).
HIGH
ComfyUI Status Indicator Hardcoded as "Online" — Always Lies
In AIStudioPanel.tsx: useEffect(() => { setComfyStatus('online') }, []). The /api/comfy-status endpoint is never called. The green status dot next to "fal.ai · ~2 сек" always shows "online" regardless of whether ComfyUI or fal.ai is actually reachable. This is a trust issue — Dan sees a false green status.
Fix: Call /api/comfy-status on mount and on a 30s interval. Display actual status. Since the app now uses fal.ai (not ComfyUI), either rename/repurpose this status check or probe fal.ai directly.
HIGH
Natali Thought Feed — Unauthenticated GET Endpoint
GET /api/nataly-thought requires no authentication. Natali's internal action log (thoughts, errors, completed actions) is publicly readable by anyone who knows the URL. This includes sensitive task context, error messages, and operational details about the agent system.
Fix: Add the same x-collector-secret / Bearer token check to the GET handler, or serve thought data only through the existing authenticated /api/activity route.
HIGH
8 Parallel Polling Hooks on Dashboard Open — Supabase Rate Risk
When DashboardPanel is open: useThoughts (5s) + useProcesses (10s) + useTasks (10s) + useNataliStatus (15s) + useMetrics (30s) + useServices (30s) + useActivity (30s) + useAgents (30s) + useWorkerStats (60s). That's a minimum of ~18 Supabase REST requests per minute per open session. With 2+ concurrent Telegram sessions (Dan on multiple devices), this compounds quickly. Supabase free tier has a 500 requests/sec limit but the pattern also generates unnecessary load and latency.
Fix: Consolidate the Status tab into a single /api/dashboard endpoint that aggregates metrics + services + natali-status + activity in one query. Replace independent SWR hooks with a shared cache key. Minimum viable: merge useActivity and useProcesses since they read the same table.
7. Medium Issues
MEDIUM
Drag-and-Drop Does Not Persist Position Reordering
The handleDragEnd in TaskBoard only calls moveTask() when the task moves to a different column. When a task is reordered within the same column, the over-task's column is compared and found equal — so nothing is saved. The visual order updates locally (React state) but resets to server order on the next 10-second SWR refresh.
Fix: On same-column drop, calculate new position and call updateTask(id, { position: newPosition }). Use fractional positioning (Lexorank / midpoint) to avoid re-numbering all tasks.
MEDIUM
Task Priority Field Is Stored But Never Displayed
All tasks have a priority field (1–5, default 2) in the database. The Task type includes it. But TaskCard.tsx and TaskColumn.tsx never render it. Dan cannot see which tasks are urgent vs. low priority on the board.
MEDIUM
No Tasks Board Link in Main Navigation Header
The /tasks page exists and is fully functional, but the main Header component has no link to it. Access requires knowing the URL or finding a link buried inside DashboardPanel (if one exists there). This is a discoverability problem that means Dan may not use task management from mobile.
MEDIUM
Duplicate Terminal Code (TerminalPanel vs. /terminal Page)
TerminalPanel.tsx and terminal/page.tsx are 90% identical — same xterm.js setup, same WebSocket logic, same ping interval, same reconnect button. They differ only in styling context (panel vs. full page). Any bug fix or feature addition must be done twice.
Fix: Extract a shared XTermWidget component and use it in both contexts.
MEDIUM
Inconsistent Supabase Client Instantiation
All API routes except agent-history/route.ts create a createClient() instance inline. agent-history imports the shared supabase from /lib/supabase.ts, which uses the anon key instead of the service role key (the lib file only exports a single client). If the shared client ever needs middleware (e.g., request interceptors), only one route benefits.
Fix: Standardize on a single pattern: either all routes use the shared client (update lib/supabase.ts to export a service-role client), or all routes instantiate inline.
MEDIUM
Token Counts Are Estimates, Not Actuals — Cost Calculation Is Wrong
import-agent-history.py uses tokens_est = size // 4 (4 bytes per token). This is wrong for files with Unicode content (common in multilingual conversations), images in tool results, or compressed formats. Actual token counts from the Anthropic API are available in OpenClaw session data but are only used in the collector path (not the import-history path). The token leaderboard in the Stats tab mixes real token counts (from OpenClaw sessions) with estimated counts (from subagent file scanning), making comparisons meaningless.
MEDIUM
set -euo pipefail Makes Collector Fragile
The -e flag causes any failing command to silently exit. If openclaw sessions exits with non-zero (timeout, process not running), the script aborts before sending metrics. The entire dashboard goes stale after 5 minutes (collector_alive = false) due to a transient openclaw glitch, not an actual system failure. Error is not logged anywhere.
Fix: Use || true on commands that are allowed to fail gracefully, and log errors to /tmp/collector.log. The openclaw call in particular should be wrapped: PARSED=$(openclaw sessions ... || echo '{"sessions":[]}').
